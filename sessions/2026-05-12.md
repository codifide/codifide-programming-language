## What Shape A means

**Thesis.** Codifide's correctness claim is verified. Now its
performance and concurrency claims need to be.

**Scope (from `dispatches/2026-05-11-v2-thinking.readout.md`):**

1. **Rust interpreter port.** The whole evaluator, not just the
   canonical form. Python reference stays as the spec; Rust becomes
   the production runtime. Expected 10-100× speedup on realistic
   programs.

2. **Graph-native parallel evaluator.** Today's tree-walker evaluates
   sequentially even when the dataflow graph has independent branches.
   A real implementation parallelizes at the node level. This is in
   the original seven design principles ("parallelism is default;
   sequencing is declared") and v1 does not deliver on it.

3. **Effect-scoped concurrency model.** If two branches share an
   `io.stdout` effect, they must serialize; if their effect sets are
   disjoint, they can run in parallel. The effect algebra already
   tells us what is safe; the runtime just has to honor it.

4. **Benchmarks as first-class citizens.** Every example gets a
   performance fixture. Regressions are regressions.

**Estimated scope:** 2-3 sessions.

**Biggest risk:** porting a moving target. Mitigation: the spec is
now written, audited, and stable. The target is less moving than
when we first deferred this work.

## What the Rust interpreter needs to implement

The Python interpreter in `codifide/runtime/interpreter.py` is the
reference. The Rust port must match it on:

- Candidate dispatch (including cost-based selection).
- Effect enforcement (primitive-call half + transitive half).
- Contract evaluation (pre/post/guard with empty effect budget).
- Belief dispatch (`believe` blocks).
- Inline conditional (`if ... then ... else`, short-circuit).
- First-class refusal (`bottom`, `RefusalError`).
- Recursion limit (default 64, typed `RecursionLimitError`).
- All eight typed error classes.
- Import resolution through the symbol store.

The graph-native parallel evaluator is a second step after the
tree-walker port passes all tests. Don't parallelize until the
sequential semantics are confirmed correct.

## Addendum — parallel evaluator + new examples (same day)

### Parallel evaluator

Filed proposal + Sable audit (conditional pass, PE-1 and PE-3
blocking findings addressed in implementation).

**What was built:**
- `crates/codifide-interpreter/src/parallel.rs` — static effect
  analysis (`expr_effects`), disjointness check, threshold check
  (`should_parallelize`), `eval_parallel_exprs` with declaration-
  order trace merge (PE-1), per-branch interpreter with inherited
  depth (PE-3).
- `call_with_vals` in `interpreter.rs` — parallel-path entry point
  for pre-evaluated args.
- Rayon added as a dependency.

**What the benchmarks revealed:**
The current programs (fizzbuzz: 29 µs, pipeline: 17 µs) are too
small for parallelism to help. Rayon's thread-spawn overhead
(~5-10 µs per task) exceeds the work in each branch. When the
parallel path was enabled on `list(fizzbuzz_one(1), ...,
fizzbuzz_one(15))`, fizzbuzz went from 29 µs to 66 µs — 2× slower.

The parallel evaluator infrastructure is correct (70/70 conformance
tests pass) and in place. The `Call` eval arm uses sequential
evaluation for now; the parallel path is available via
`eval_parallel_exprs` and `should_parallelize` for programs with
branches that each take >100 µs.

**Honest v2-A summary:** The sequential Rust interpreter is 6–25×
faster than Python. The parallel evaluator is architecturally
correct but needs larger programs to demonstrate a speedup.

### Three new example programs

1. **`examples/batch_classify.cod`** — batch image classification.
   Eight independent `safe_classify` calls, each using `believe`
   dispatch with a confidence threshold. Demonstrates the primary
   parallel evaluator use case (eight independent model calls).
   Results: `['cat', 'dog', 'uncertain', 'fish', 'uncertain',
   'rabbit', 'uncertain', 'turtle']`.

2. **`examples/recursive_sum.cod`** — recursive list sum with a
   postcondition that cross-checks against the `sum` primitive.
   Demonstrates the cost-dispatch idiom for recursive functions
   (base case cost 1, recursive case cost 10). Results:
   `[15, 60, 0, 100]`.

3. **`examples/text_stats.cod`** — multi-statistic text analysis.
   Computes word count, char count, question detection, and length
   classification for three sentences. Demonstrates multiple
   independent pure functions composed into a result list — the
   parallel evaluator opportunity for larger programs. Results:
   `[[2, 10, False, 'short'], [9, 35, False, 'medium'],
   [4, 16, True, 'short']]`.

All three pass on both Python and Rust runtimes with identical
results.
