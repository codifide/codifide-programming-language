# Sable — Python CLI filesystem-safety audit (2026-05-11)

Auditor: Sable
Scope: `codifide/__main__.py` file-read paths. The store was hardened on
2026-05-10 (P1-5 symlink escape, TOCTOU defense). The Python CLI's
input-reading surface had not been audited in the same pass. This
audit probes it.

## Findings

### AUD-2026-05-11-05 — P1 — unbounded source read

**What.** `_read(path)` used `Path(path).read_text(encoding="utf-8")`
with no size bound. Reading from `/dev/zero` returned only when
memory was exhausted. Reading a 50 MiB junk file loaded 50 MiB into
memory before the parser rejected it.

**Probe.**

```
$ python3 -m codifide canonical /dev/zero
# hangs indefinitely until OOM
```

Same class of bug as audit P1-7 (the Rust canonical binary's
`fs::read_to_string` hang on `/dev/zero`, fixed on 2026-05-10).
The Python CLI's file-read path was not touched in that pass.

**Why it matters.** A Codifide program is untrusted input. The CLI is
how hosts feed programs to the runtime. An unbounded read lets a
hostile or accidental input exhaust the embedding process's memory
before a single typed error is raised. The typed-error discipline
the language commits to assumes the file fits; that assumption was
unstated.

**Severity.** P1. Not data loss, not escape. But the runtime's
soundness rests on "a typed error is the worst outcome," and an
unbounded read violates that in the most trivial way.

**Fix (now applied).** `_read` uses `open(...).read(cap + 1)` with
`_MAX_SOURCE_BYTES = 16 MiB`. Excess raises `ParseError` with a
specific message. A non-UTF-8 body raises `ParseError` rather than
leaking `UnicodeDecodeError`. Regression tests in
`tests/test_cli_filesystem.py`.

### AUD-2026-05-11-06 — P3 — symlink followed by `codifide run`

**What.** The CLI happily follows symlinks to files outside the
working directory. `codifide run <link-to-/etc/passwd>` reads the
target and reports a parse error citing the target's contents.

**Probe.**

```
$ ln -s /etc/passwd link.cod
$ python3 -m codifide run link.cod
codifide: unexpected top-level line: 'nobody:*:-2:-2:Unprivileged User:...'
```

**Why it matters.** This is not a vulnerability on its own — UNIX
file permissions gate the read — but it is a reliable oracle for
what the invoking user can read.

**Severity.** P3. The store audit rejected symlinks because the
store controls its namespace and must defend against adversarial
path planting. The CLI reads arbitrary user-supplied paths; there
is no analogous invariant to protect. Refusing symlinks would
break legitimate developer workflow (symlinked source files in a
shared directory) without a clear safety win.

**Fix.** Accepted as documented CLI behavior. No change.

### AUD-2026-05-11-07 — P3 — `--store` accepts path-traversal-style roots

**What.** The store subcommands accept any path the user supplies
via `--store`. A value like `../../../tmp/evil_store` is used
verbatim.

**Probe.**

```
$ python3 -m codifide store --store ../../../tmp/evil_store list
# succeeds (no objects) or fails at the directory level
```

**Why it matters.** Also not a vulnerability. The user controls the
store root; passing a relative path is a documented CLI feature.
Path traversal through this argument goes wherever the user pointed
it. The same argument shape appears in `git --git-dir=...` and is
not considered a bug.

**Severity.** P3. No change.

## What I did not test

- Concurrent writes by multiple CLI invocations to the same store.
  This was covered by `tests/test_store_concurrency.py` on the
  store layer. The CLI does no additional locking.
- The `codifide capability` subcommand's hash stability under
  environmental variation. Marginal to this audit.
- Whether `--store` accepts paths like `/dev/null` and degrades
  gracefully. Not probed; probably produces an OSError wrapped by
  StoreError, but I did not confirm.
- Windows-specific path handling. macOS-only probes.
- The `codifide verify` subcommand with a symlink-in-the-middle
  path. Should behave the same as `run`; I did not isolate it.

- AUD-06 (P3) and AUD-07 (P3) are accepted as documented behavior;
