## Where we are

v2.0 shipped. The adoption story is now clean:

- 5/5 first-attempt successes in the Relay v2.0 case study (this session)
- Prior baseline: 3–4/5 across four models
- The three changes that drove the improvement: confidence threshold fix,
  transitive dependency documentation, double-print documentation
- AUD-OVERNIGHT-02 (parallel evaluator import gap): P3, unreachable by
  construction, fix path known, deferred to v3.0 parallel work

The v2.0 thinking doc (2026-05-11) recommended Shape A → Shape B → Shape C.
Shape A shipped (Rust interpreter, parallel evaluator, benchmarks). v3.0 is
Shape B: the protocol agents actually exchange symbols over.

---

## v3.0 thesis

**Codifide is currently a solo-agent language.** An agent can write programs,
publish them to a local store, and import them by hash — but only within one
filesystem. Two agents on different machines cannot exchange symbols without
out-of-band coordination. The content-addressed store is the right foundation
for a protocol; v3.0 builds the protocol on top of it.

The RPC API (V2-1) is the first step: it proved that the canonical form can
travel over HTTP and that hash-verified retrieval works. v3.0 extends this
from a local server to a network-accessible exchange.

## v3.0 requirements

### V3-1 — Parallel evaluator: full import support (P1)

**Evidence:** AUD-OVERNIGHT-02. The parallel evaluator's threshold currently
excludes imported-symbol calls. This is safe (the gap is unreachable) but
it means the parallel evaluator cannot fire on the programs that matter most
for agent use cases — pipelines composed from content-addressed symbols.

**What it is:** Relax the `should_parallelize` threshold to support imported-
symbol calls. Pass `resolved_imports` to branch interpreters. Add the
regression test specified in AUD-OVERNIGHT-02.

**Acceptance criterion:** A program that imports two pure symbols and calls
them inside `list(f(x), g(x))` evaluates both in parallel. The regression
test passes.

**Dependency:** None. This is the first v3.0 item because it unblocks the
parallel evaluator for real workloads.

---

### V3-2 — Remote symbol resolution (P1)

**Evidence:** The v2.0 thinking doc (Shape B). The RPC API (V2-1) proved
the local HTTP pattern. The next step is making content identities resolvable
across machines.

**What it is:** A simple HTTP protocol for symbol exchange. An agent with a
content identity (`sha256:<hex>`) can resolve it against any server that
holds the symbol — not just its local store. The hash-verification path
(already in the store) makes trust automatic: a server cannot lie about a
symbol's content without changing its identity.

**Minimal form:** A public read-only endpoint at `codifide.com/symbols/<identity>`
that serves canonical CBOR for any symbol published to the public registry.
Agents can publish via the existing RPC API (`POST /symbols`) and resolve
remotely via the new endpoint.

**Acceptance criterion:** An agent on machine A publishes a symbol. An agent
on machine B resolves it by identity without any out-of-band coordination.
The hash-verification path rejects a tampered response.

**Dependency:** V2-1 (RPC API, shipped). The public endpoint is an extension
of the existing server, not a new service.

---

### V3-3 — Refusal reasons (P2)

**Evidence:** Four case studies produced programs where `bottom` escaped and
the error message was `RefusalError: main returned ⊥`. The refusal is
correct but opaque. An agent debugging a pipeline cannot tell which candidate
refused or why.

**What it is:** Structured refusal reasons. `bottom` gains an optional string
payload: `bottom "confidence below threshold"`. The `RefusalError` message
includes the reason. The canonical form gains a `reason` field on the
`bottom` AST node (optional, backward-compatible).

**Acceptance criterion:** `bottom "reason string"` parses and evaluates.
`RefusalError` includes the reason in its message. Programs without reasons
behave identically to today.

**Dependency:** None. Canonical-form extension; backward-compatible.

---

### V3-4 — Time-indexed types: `T@timestamp` (P3)

**Evidence:** Listed in the v2.0 thinking doc (Shape C) and the pre-adoption
roadmap. No agent session has produced evidence that agents need this. It
remains on the roadmap because it is architecturally coherent with the effect
algebra and because the adoption baseline is now strong enough to justify
a speculative addition.

**What it is:** A type modifier that tracks when a value was produced.
`Price@now` is not interchangeable with `Price@yesterday`. The type system
rejects stale values at call sites that require fresh ones. The effect algebra
gains `clock.at(t)` so time is a typed dependency, not a side channel.

**Acceptance criterion:** `sig (p: Price@now) -> Decision` rejects a
`Price@yesterday` argument with a typed error. `clock.at(t)` is a valid
effect declaration.

**Dependency:** V3-3 (refusal reasons) — a stale-value rejection should
produce a structured refusal, not a bare error.

**Caveat:** This is the only v3.0 item without direct adoption evidence.
It should be the last item implemented and should be dropped if V3-1 through
V3-3 produce evidence that agents don't need it.

## What is explicitly deferred

**Hosted runtime / cloud service:** No adoption evidence that agents need
managed hosting. The public symbol endpoint (V3-2) is the right first step;
a full hosted runtime is premature.

**Certification program:** Premature without external implementors.

**Moltbook integration:** Depends on V3-2 (remote symbol resolution). Not
scoped until V3-2 ships.

**Cost as an expression (`cost len(xs)`):** No adoption evidence. Deferred.

**Structural diff and merge:** No adoption evidence. Deferred.

**Effect inference:** No adoption evidence. Deferred.

---

## Sequencing

1. **V3-1** (parallel import support) — unblocks the parallel evaluator for
   real workloads. Small scope, known fix path. Do first.
2. **V3-2** (remote symbol resolution) — the core v3.0 value proposition.
   Requires design dispatch before implementation.
3. **V3-3** (refusal reasons) — canonical-form change; land before V3-4
   because V3-4 depends on it.
4. **V3-4** (time-indexed types) — last, and conditional on evidence.

---

## What v3.0 proves

That Codifide is a multi-agent protocol, not just a solo-agent language.
An agent that publishes a symbol to the public registry and another agent
that imports it by hash — without any out-of-band coordination — is the
proof of concept. V3-2 is the item that delivers it.
