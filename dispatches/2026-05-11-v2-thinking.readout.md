# v2 thinking — three honest shapes (2026-05-11)

*By Quill, at session close.*

v1 shipped. User asked what v2 looks like. This is the thinking,
documented for the next session to pick up from. Nothing here is
a decision yet — the user is thinking about it.

## What v1 actually is

v1 is a language whose canonical form is stable and
cross-implementation, whose runtime semantics are typed and
auditable, and whose authoring surface has been smoothed enough
that an agent reading the manifest can write correct programs.
It does what it says on the tin.

What it is not yet:

- **A runtime agents use at scale.** The interpreter is a
  tree-walker in Python. Fine for the dozens of test programs.
  Not fine for a swarm of agents exchanging thousands of symbols
  per second.
- **A protocol.** The canonical form is byte-stable, but there is
  no defined way for two agents to actually exchange symbols
  across a network. Everything flows through the local filesystem
  store.
- **A social thing.** Codifide is stewarded by two people
  (Douglas + Claude) and used by zero external agents in
  production.

v2 is whichever of those three matters most.

## Three coherent v2 shapes

Each is internally consistent. You would pick based on what you
want Codifide to prove next.

### Shape A — "the runtime agents can actually use"

**Thesis.** Codifide's correctness claim is verified; now its
performance and concurrency claims need to be.

**What v2 does.**

- **Rust interpreter port.** Not just the canonical form — the
  whole evaluator. Python reference stays as the spec; Rust
  becomes the production runtime. Expect 10-100× speedup on
  realistic programs.
- **Graph-native parallel evaluator.** Today's tree-walker
  evaluates sequentially even when the dataflow graph has
  independent branches. A real implementation parallelizes at
  the node level — this is in the original seven principles
  ("parallelism is default; sequencing is declared") and v1
  does not deliver on it.
- **Effect-scoped concurrency model.** If two branches share an
  `io.stdout` effect, they must serialize; if their effect sets
  are disjoint, they can run in parallel. The effect algebra
  already tells us what is safe; the runtime just has to honor
  it.
- **Benchmarks as first-class citizens.** Every example gets a
  performance fixture. Regressions are regressions.

**Rough scope.** 2-3 solid sessions.

**Biggest risk.** Porting a moving target. Mitigation: the spec is
now written and audited, so the target is less moving than when
we first deferred this work.

**What this shape proves.** Codifide is fast enough to actually
use. The seven design principles include "parallelism is default";
v2-A makes that real.

### Shape B — "the protocol agents actually exchange symbols over"

**Thesis.** Content addressing is useful only if agents can
resolve content addresses across a network, not just across a
local directory.

**What v2 does.**

- **Remote symbol resolution.** An identity `sha256:<hex>`
  resolves through a content-addressed network — start with a
  simple HTTP fetch mapping identity → bytes, with the existing
  hash-verification path making trust automatic. No central
  authority; any agent can serve its own store.
- **Signed dispatches as an exchange format.** Two agents
  communicating "I computed this and believe it with confidence
  0.7; my store identity is X" is a protocol message. The Glyph
  dispatch schema already has the right shape; v2 makes it a
  wire message with canonical bytes.
- **Capability-mediated imports.** Today's
  `import name = sha256:...` binds content directly. v2 adds
  capability tokens — an agent says "I can resolve this
  identity if you give me capability X," making shared stores
  feasible without shared filesystems.
- **A reference network.** Not blockchain; something closer to
  IPFS-for-symbols — but with Codifide's specific
  effect-transitivity semantics enforced at resolve time (an
  import that claims `effects {}` really is pure when I fetch
  it, because the hash covers the effect set).

**Rough scope.** 3-4 sessions.

**Biggest risk.** "The network" is a big scope. Mitigation: start
with a local HTTP-over-filesystem stub that proves the protocol,
defer the real network.

**What this shape proves.** Codifide is actually usable *between*
agents, not just within one. This is what makes the "confidence in
code" tagline operational across organizational boundaries.

### Shape C — "the language agents genuinely prefer over alternatives"

**Thesis.** v1 is technically capable but only marginally more
expressive than Python-with-decorators. What makes v2
distinctively useful is closing the remaining expressiveness gaps
that keep agents reaching for other languages.

**What v2 does.**

- **Time-indexed types: `T@timestamp`.** Already on the roadmap.
  A `Price@now` is not interchangeable with a `Price@yesterday`;
  the type system tracks staleness. The effect algebra gains
  `clock.at(t)` so time is a typed dependency, not a side
  channel.
- **Cost as an expression, not a literal.** v1's cost is a
  static integer. v2 lets cost depend on input —
  `cost len(xs)` makes a candidate's priority scale with its
  input size, so the dispatcher can prefer cheap candidates for
  big inputs.
- **Structural diff and merge.** Two modules that differ only in
  candidate declaration order, intent wording variations, or
  cost annotations can be merged programmatically. This is the
  "agents collaborating on code" shape that today requires
  human-style text diffs.
- **A type system.** Not Hindley-Milner; something lighter —
  type inference for literals and effect inference for candidate
  bodies, so agents can elide `sig` declarations and let the
  compiler fill them in, then re-expand to full form on publish.
- **Refusal reasons.** Today `bottom` is opaque. v2 gives refusal
  a structured reason: `bottom "confidence below threshold"`
  makes the refusal auditable without breaking the
  first-class-refusal property.

**Rough scope.** 4-5 sessions across multiple proposals.

**Biggest risk.** Feature creep disguised as expressiveness.
Mitigation: every feature needs an observed gap to justify it. No
"would be nice" additions.

**What this shape proves.** Codifide is genuinely the right tool
for agentic coding, not just a technically-correct experiment.

## Recommendation (from Claude)

**Shape A first, then Shape B, then Shape C.**

Reasoning:

- **A before B** because shipping a protocol over a slow runtime
  produces a slow network. Performance work first means the
  protocol we design later can afford to include semantic checks
  at the wire that would not be affordable in tree-walker-land.
- **B before C** because expressiveness without distribution is a
  solo-agent feature, and Codifide's distinctive value proposition
  is multi-agent. Adding structural diff before there is a
  protocol to diff over is premature.
- **C as the finish** because by the time the runtime is fast and
  the protocol works, we will have real external users telling us
  which expressiveness gaps actually hurt. The current list is
  Claude's guess; theirs would be better.

**Alternative sequence** (also defensible): **C first**, because
the time-indexed types and refusal reasons change the canonical
form, and it is better to land canonical-form changes in a batch
rather than dribble them out across three versions.

## The real thing to decide first

**The thing that would most change this ranking is a real
external-model run.**

- If an external agent hands Codifide back saying "the syntax is
  fine but everything's slow," v2 is A.
- If they say "I can't use this without a way to share what I've
  built with other agents," v2 is B.
- If they say "I keep reaching for things the language can't
  express," v2 is C.

The single most valuable thing between v1 and v2 is probably not
any of these three shapes — it is the external-model experiment
that tells us which shape is the right one to start.

## Claude's actual call (restated)

Run the external-model experiment as a pre-v2 investigation. Then
spend v2 on whichever shape the experiment names. The experiment
is not a v2 feature; it is a v2 targeting decision. Skipping it is
how we would build the wrong v2 confidently.
