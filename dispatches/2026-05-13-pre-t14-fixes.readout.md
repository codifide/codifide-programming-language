# Pre-T1-4 Fixes — Three Findings from T1-2 and T1-3

**Date:** 2026-05-13

## What happened

Before running the Claude baseline session (T1-4), three open findings from
the GPT-4o and Gemini case studies were reviewed by all three personas and
applied by consensus.

---

## Finding 4 — `lower()` omission (GEM-2, P3)

**Problem:** `contains()` is case-sensitive. Gemini's classifier used
`contains(message, "spam")` without normalizing first. This works for
lowercase test messages but fails silently on mixed-case input like
`"This Message Contains SPAM"`.

**Fix applied:**
- `docs/AGENT_QUICKREF.md` — added case-sensitivity note to the Strings
  section: *"`contains` is case-sensitive. Normalize first:
  `contains(lower(msg), "keyword")`"*
- `docs/AGENT_TASK_SPEC.md` — added `lower()` reminder to the Program 1
  requirements
- `docs/GPT4O_PROMPT.md` — same note added to the prompt's quickref section

---

## Finding 5 — `is_bottom()` trap (GEM-1, P2)

**Problem:** `is_bottom()` is a value inspector, not a propagation catcher.
It returns `true` when passed a literal `bottom` value. It cannot catch a
`bottom` that propagated through a bind — `bottom` raises
`BottomPropagationError` before `is_bottom()` can inspect it. Gemini used
`is_bottom()` defensively in `route_message`, reasoning it would catch
refusals from `moderate`. That code is dead — it never executes.

Confirmed by test: calling `is_bottom()` on the result of a function that
returns `bottom` raises `BottomPropagationError: primitive 'is_bottom'
received ⊥`.

**Fix applied:**
- `docs/AGENT_QUICKREF.md` — added `is_bottom` idiom section with working
  and broken examples, explanation of when it's useful
- `docs/AGENT_QUICKREF.md` — updated the `bottom` propagates rule to
  cross-reference the `is_bottom` note
- `docs/GPT4O_PROMPT.md` — added `is_bottom` section to the prompt

---

## Finding 6 — Routing style undocumented (GEM-3, P3)

**Problem:** Both models produced working routing code but used different
patterns — GPT-4o used multiple `cand` + `when` guards; Gemini used nested
`if/then/else` in a single candidate body. Neither is wrong, but the docs
were silent on which is preferred and when.

**Fix applied:**
- `docs/AGENT_QUICKREF.md` — added "Multi-branch value routing" idiom
  section documenting both patterns with guidance: `cand` + `when` for
  three or more branches (idiomatic, shows intent per branch); `if/then/else`
  for binary choices inside a body
- `docs/GPT4O_PROMPT.md` — same guidance added to the prompt

---

## Assessment

All three fixes are documentation-only — no language changes, no spec
amendments, no new primitives. The `is_bottom()` finding is the most
significant: it's a genuine trap that any agent reading the manifest would
fall into. The routing style guidance closes an ambiguity that produced
divergent but correct code across two sessions. The `lower()` note is
defensive hygiene.

T1-4 (Claude baseline) now runs against a cleaner set of docs. If Claude
independently reaches for `is_bottom()` as a propagation catcher, that's
evidence the fix wasn't sufficient. If Claude uses `lower()` without being
told, that's evidence the fix was unnecessary for this model.

What I'm not yet sure of: whether there are other `is_bottom()`-class traps
in the manifest — primitives that look like they handle `bottom` but actually
can't. A Sable audit of the full primitive surface for `bottom`-adjacent
behavior would be worth filing after T1-4.
